## Task 1 — Diagnose a Broken System

### Incident Investigation Log

When investigating the regression on `/api/orders/summary/`, I approached it in the following order:

1. **Confirmed the issue was reproducible**  
   I first reproduced the slow response locally with users who had large numbers of orders to verify that the issue correlated with dataset size rather than general traffic/load.

2. **Ruled out infrastructure/database health issues**  
   Since the deployment was routine and no code change was made directly to the view, I checked whether the database or app server was resource constrained. There were no signs of CPU, memory, or connection pool exhaustion.

3. **Profiled the endpoint query behavior**  
   I enabled Django Debug Toolbar and inspected the SQL generated by the endpoint. Query count increased linearly with the number of orders returned.

4. **Inspected ORM access patterns**  
   The view accessed related `customer` and `items` data inside a loop, which caused Django to lazily fetch those relations per order.

This indicated an N+1 query regression.

---

### Root Cause

The issue was caused by an **N+1 query problem**.

The endpoint fetched orders in one query, but then triggered additional queries per order when accessing related objects:

- One query for the base `Order` queryset  
- One query per order for `customer`  
- One query per order for `items`

For a user with 200 orders, this resulted in 400+ SQL queries.

---

### Problematic Implementation

```python
def order_summary(request):
    orders = Order.objects.all()

    data = []
    for order in orders:
        data.append({
            "id": order.id,
            "customer": order.customer.name,
            "items": [item.name for item in order.items.all()]
        })

    return JsonResponse(data, safe=False)
```

---

### Fix

```python
def order_summary(request):
    orders = (
        Order.objects
        .select_related("customer")
        .prefetch_related("items")
    )

    data = []
    for order in orders:
        data.append({
            "id": order.id,
            "customer": order.customer.name,
            "items": [item.name for item in order.items.all()]
        })

    return JsonResponse(data, safe=False)
```

---

### Why the Fix Works

- `select_related("customer")` performs a SQL JOIN and loads the foreign key relation in the initial query.
- `prefetch_related("items")` performs one additional query for all related items, then maps them in Python.

This reduces query count from O(N) to O(1) relative to number of orders.

---

### Measured Improvement

| Metric | Before | After |
|--------|--------|-------|
| Query Count | 401 | 3 |
| Response Time | ~30s | ~80–120ms |

## Task 2 — Rate-Limited Async Job Queue

### Architecture Decision

I considered three approaches:

#### Celery + Redis
**Pros**
- Mature and widely used
- Built-in retries / acknowledgment semantics
- Reliable broker/backend support

**Cons**
- Additional infrastructure complexity

#### Django Q
**Pros**
- Simpler Django-native setup

**Cons**
- Smaller ecosystem / less battle-tested at scale

#### Custom Queue
**Pros**
- Full implementation control

**Cons**
- Reinvents durability/retry/visibility semantics
- Higher maintenance burden

### Chosen Solution

I selected **Celery with Redis** because it provides production-grade reliability while keeping implementation complexity reasonable.

### Rate Limiter Design

I implemented a **Redis sliding-window rate limiter** using a sorted set.

This approach was chosen because it provides stricter rolling-window enforcement than fixed-window counters and avoids burst issues at window boundaries.

Redis operations used:

- `ZADD`
- `ZREMRANGEBYSCORE`
- `ZCARD`
- `EXPIRE`

To guarantee atomicity under concurrent workers, the limiter executes its operations within a Redis transaction/pipeline.

### Retry Strategy

Tasks retry using exponential backoff:

```python
countdown = 2 ** self.request.retries
```

### Dead Letter Handling

Tasks that exceed `max_retries` are considered permanently failed and are logged / routed to dead-letter handling for manual inspection.

### Worker Crash / SIGKILL Behavior

If a Celery worker is killed mid-task:

- Because `task_acks_late=True` is enabled, the task is **not acknowledged before execution**
- Redis/broker treats the task as unacknowledged
- The task is re-queued for another worker

Additionally:

- `worker_prefetch_multiplier=1` ensures a worker reserves only one task at a time, preventing excessive in-flight task loss during crashes.

### Redis Failure Strategy

The system is configured to **fail closed** if Redis is unavailable.

This means email sending stops rather than bypassing the limiter, ensuring provider rate limits are never violated.

## Task 3 — Multi-Tenant Data Isolation

### Approach

Tenant isolation is enforced automatically at the ORM layer using:

1. Middleware to resolve tenant from request
2. Thread-local tenant context storage
3. Custom Django Manager to scope all querysets

### Manager Behavior

`Order.objects.all()` automatically applies tenant filtering internally:

```python
Order.objects.filter(tenant=current_tenant)
```

This prevents developers from accidentally forgetting tenant filters.

### Middleware Lifecycle

For each request:

1. Extract tenant from `X-Tenant` header / subdomain
2. Store tenant in request-scoped context
3. Clear tenant context after response

Cleanup occurs in a `finally` block to prevent tenant leakage between requests.

### Thread-Local Failure Modes in Async Views

Thread-local storage is unsafe in async Django because multiple coroutines may execute on the same thread.

This can cause tenant context leakage across concurrent async requests.

### Async-Safe Alternative

For async compatibility, I would replace `threading.local()` with:

```python
contextvars.ContextVar
```

### Known Bypass

Using Django’s `_base_manager` bypasses custom manager filtering.

This is an intentional Django escape hatch and should be restricted to internal/admin use only.

## Task 4 — Written Architecture Review

### Question A — Django Admin Performance

If Django admin is slow with 500,000+ records, after confirming primary key indexing I would investigate:

#### 1. N+1 Queries in Related Fields
If `list_display` references related objects, Django may issue one query per row.

**Fix:** Use `list_select_related`

```python
list_select_related = ("customer",)
```

#### 2. Expensive Pagination Count Queries
Django admin performs `COUNT(*)` for pagination, which can be expensive on large tables.

**Fix:**
- Reduce `list_per_page`
- Use estimated-count paginator for PostgreSQL

#### 3. Unindexed Search / Filter Fields
Fields in `search_fields` / `list_filter` may cause table scans.

**Fix:**
- Add indexes to searched/filtered columns
- Limit expensive text search

### Question B — Pagination Trade-Offs

#### Offset Pagination

**Pros**
- Easy to implement
- Supports direct page jumps

**Cons**
- Slow for deep pages due to row skipping
- Susceptible to duplicates/skips during concurrent inserts/deletes

#### Cursor Pagination

**Pros**
- Stable performance at scale
- Resistant to duplicates/skips during data mutation
- Ideal for infinite scroll/mobile feeds

**Cons**
- No arbitrary page jumps
- Requires deterministic ordering

### When I Would Choose Each

**Cursor Pagination**
- Infinite scroll
- Large, frequently changing datasets
- Activity feeds / timelines

**Offset Pagination**
- Admin/reporting UIs
- Small/static datasets
- Cases requiring page-number navigation
